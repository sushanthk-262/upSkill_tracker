Blockchain Fundamentals

Weeks 1–2

What is a Blockchain?

A distributed ledger of transactions, cryptographically linked, stored across many nodes. No single party controls it.

  • Decentralized: No single point of control or failure
  • Immutable: Once confirmed, data cannot be altered
  • Transparent: Anyone can verify any transaction
  • Trustless: Cryptography replaces trust in counterparties
  • Permissionless: Anyone can transact without approval

Hashing — The Foundation

One-way function: any input → fixed-size output. Deterministic, collision-resistant, irreversible.

Input:  "Hello World"
SHA-256: a591a6d40bf420404a...

Input:  "Hello World!"  (one char diff)
SHA-256: 03675ac53ff9cd1535...
  • SHA-256: Bitcoin — 256-bit output
  • Keccak-256: Ethereum — 256-bit output
  • Ed25519: Solana signatures (faster)

Public-Key Cryptography

Private Key (256 random bits — keep secret!)
        ↓  secp256k1 (ETH) / Ed25519 (SOL)
Public Key (derived, share freely)
        ↓  hash + encode
Address   (shorter, what others send funds to)

ETH address:  0x742d35Cc6634C0532...  (40 hex)
SOL address:  7Np41oeYqPefeNQh...    (Base58)

Digital Signatures

Prove you own a private key without revealing it.

Sign:   hash(message) + private_key → Signature
Verify: Signature + message + public_key → ✅/❌

Ethereum: ECDSA on secp256k1 curve
Solana:   EdDSA on Ed25519 curve (smaller, faster)

# Why it matters:
# Every tx is signed — proves sender authorised it
# Validators verify before including in block

HD Wallets (BIP-39 / BIP-44)

One 12/24-word seed → unlimited recoverable addresses.

Mnemonic phrase (BIP-39):
"abandon ability able about above..."

        ↓ PBKDF2 (2048 iterations)
512-bit seed
        ↓ BIP-44 derivation
m/44'/60'/0'/0/0  → ETH address 0
m/44'/60'/0'/0/1  → ETH address 1
m/44'/501'/0'/0'  → SOL address

# Lose seed phrase = lose everything. No recovery.

Proof of History: verifiable delay function that proves time elapsed between events — eliminates need for validators to communicate to agree on ordering.

Merkle Trees

Binary tree of hashes. Root hash summarises all transactions. Used for efficient proof of inclusion.

       Root Hash
      /          \
   Hash(AB)    Hash(CD)
   /    \       /    \
 H(A)  H(B)  H(C)  H(D)
  A     B     C     D  (transactions)

# Proof that A is in the tree:
# → provide H(B) and Hash(CD)
# → verifier reconstructs Root Hash
# Used by: light clients, NFT whitelists (Merkle proofs)

Block Structure

Block {
  header: {
    parentHash,          // Links to previous block
    stateRoot,           // Merkle root of all accounts
    transactionsRoot,    // Merkle root of all txns
    receiptsRoot,        // Merkle root of tx receipts
    timestamp,
    number,
    nonce (PoW) / validatorSig (PoS)
  },
  transactions: [tx1, tx2, ...]
}
# Changing any tx → changes transactionsRoot
# → changes header hash
# → invalidates all subsequent blocks (immutability)

Ethereum & EVM

Weeks 1–4

ETH Units & Gas

1 ETH = 10^18 wei
1 Gwei = 10^9 wei  (gas prices quoted here)

Fee = Gas Used × (Base Fee + Priority Fee)

Typical gas costs:
  ETH transfer:       21,000 gas
  ERC20 transfer:    ~50,000 gas
  ERC721 mint:       ~70,000 gas
  Uniswap swap:     ~100,000–150,000 gas
  Contract deploy:  200,000–2,000,000 gas

Account Types & State

  • EOA (Externally Owned): Private key → address. Initiates txns. No code.
  • Contract Account: No private key. Has code + storage. Cannot initiate, only react.
Account state:
{
  nonce:    tx count (EOA) or deployments (contract)
  balance:  wei held
  codeHash: hash of contract bytecode (empty for EOA)
  storageRoot: Merkle root of contract storage
}

EVM — How Smart Contracts Execute

  • Stack-based virtual machine — 256-bit word size
  • Solidity compiles to EVM bytecode
  • Every opcode has a gas cost (SLOAD = 2100 gas, SSTORE = 20,000)
  • Storage is expensive — memory/calldata is cheap
  • All EVM-compatible chains (Polygon, Arbitrum, BSC) run same bytecode
Key insight: This is why your Intel OS knowledge helps. EVM is an OS running inside another OS — understanding virtual machines, memory models, and execution contexts is directly applicable.

Transaction Lifecycle

1. Create tx { to, value, data, gas, nonce }
2. Sign with private key → get signature (v, r, s)
3. Broadcast to mempool (public pending pool)
4. Validator selects (highest fee priority)
5. Execute in EVM — success or revert
6. Include in block → propagate to network
7. Subsequent blocks → "confirmations"
8. Finality: ~12s (slot) / ~13min (checkpoint)

Nonce: sequential counter → prevents replay attacks
chainId: prevents cross-chain replay (EIP-155)

Security: Reentrancy Attack + Fix

// ❌ VULNERABLE
function withdraw() public {
    uint amt = balances[msg.sender];
    (bool ok,) = msg.sender.call{value: amt}(""); // external call FIRST
    balances[msg.sender] = 0;  // state update AFTER → attacker re-enters!
}

// ✅ SECURE — Checks-Effects-Interactions
function withdraw() public nonReentrant {
    uint amt = balances[msg.sender];
    balances[msg.sender] = 0;  // 1. Update state FIRST
    (bool ok,) = msg.sender.call{value: amt}(""); // 2. Then external call
    require(ok);
}
// Also use: OpenZeppelin ReentrancyGuard

    require(!minted[msg.sender], "Already minted");

AMM Pricing (x * y = k)

No order book. Price set by reserves ratio.

Pool: 100 ETH + 200,000 USDC  →  k = 20,000,000
Price of ETH = 200,000 / 100 = $2,000

User swaps 1 ETH in:
  new_x = 101
  new_y = k / new_x = 20,000,000 / 101 ≈ 198,020
  USDC out = 200,000 - 198,020 ≈ 1,980 USDC
  (slightly less than $2000 = price impact)

Larger trade → more price impact (slippage)

Account Model — The Critical Difference

Account = {
  lamports:   u64,      // SOL balance (1 SOL = 10^9)

Sealevel — Parallel Execution

  • Transactions declare all accounts they'll read/write upfront
  • Sealevel scheduler runs non-overlapping txns in parallel
  • Txns touching different accounts = run simultaneously
  • This is why Solana achieves high TPS vs sequential EVM
  • As developer: declare accounts precisely → enables parallelism
Intel connection: This is parallel execution scheduling — same concepts as multi-core CPU dispatch. Your systems background gives you intuition here that most Web3 devs lack.

### Standard — O(log n) def binary_search(arr, target): left, right = 0, len(arr) - 1 while left <= right: mid = (left + right) // 2 if arr[mid] == target: return mid elif arr[mid] < target: left = mid + 1 else: right = mid - 1 return -1 ### Search in Rotated Array def search(nums, target): l, r = 0, len(nums) - 1 while l <= r: mid = (l + r) // 2 if nums[mid] == target: return mid if nums[l] <= nums[mid]: # left half sorted if nums[l] <= target < nums[mid]: r = mid - 1 else: l = mid + 1 else: # right half sorted if nums[mid] < target <= nums[r]: l = mid + 1 else: r = mid - 1 return -1

1D DP Patterns

### Climbing Stairs (Fibonacci) def climbStairs(n): if n <= 2: return n a, b = 1, 2 for _ in range(3, n + 1): a, b = b, a + b return b ### House Robber (linear DP) def rob(nums): prev2, prev1 = 0, 0 for n in nums: prev2, prev1 = prev1, max(prev1, prev2 + n) return prev1 ### Maximum Subarray (Kadane's) def maxSubArray(nums): max_sum = cur = nums[0] for n in nums[1:]: cur = max(n, cur + n) max_sum = max(max_sum, cur) return max_sum

Backtracking Template

### Subsets — O(2^n) def subsets(nums): result = [] def backtrack(start, path): result.append(path[:]) for i in range(start, len(nums)): path.append(nums[i]) backtrack(i + 1, path) path.pop() backtrack(0, []) return result ### Combination Sum (with reuse) def combinationSum(candidates, target): result = [] def backtrack(start, path, remaining): if remaining == 0: result.append(path[:]); return for i in range(start, len(candidates)): if candidates[i] <= remaining: path.append(candidates[i]) backtrack(i, path, remaining - candidates[i]) path.pop() backtrack(0, [], target) return result

CAP Theorem

Distributed systems can guarantee only 2 of 3 simultaneously:

  • Consistency: All nodes return same data at same time
  • Availability: Every request gets a non-error response
  • Partition Tolerance: Works despite network failures between nodes
Network partitions always happen → P is non-negotiable.
CP: Bank systems — consistent over available.
AP: Social feeds, DNS — available over consistent.

URL Shortener

// bit.ly, tinyURL Components: - API: POST /shorten → returns short code - DB: { short_code, original_url, user_id, created } - Redirect: GET /{code} → 301/302 redirect Short Code Generation: Option 1: Base62(auto-increment ID) → sequential, predictable Option 2: MD5(URL)[0:8] → hash-based, possible collision Option 3: Random + DB uniqueness check Scale: - Read-heavy → cache popular URLs in Redis - TTL on cache: most URLs accessed shortly after creation - Analytics: async increment click counter via queue

Twitter Feed / News Feed

// Fan-out strategies: Fan-out on write (push): - On tweet: push to all followers' feed cache - Pro: fast reads; Con: expensive for celebrities (10M+ followers) Fan-out on read (pull): - On load: fetch tweets from all followed users - Pro: simple writes; Con: slow reads Hybrid (Twitter's actual approach): - Regular users → fan-out on write - Celebrities → fan-out on read at query time - Redis sorted sets for feed (score = timestamp) - Pagination: cursor-based (not offset)

Real-time Chat

// WhatsApp, Slack Key: persistent connections for real-time delivery Architecture: WebSocket server (stateful, sticky sessions) → Message broker (Kafka) for routing → DB (Cassandra) for message history → Push notifications for offline users Message delivery states: Sent → Delivered → Read (WhatsApp ticks) Scale: - Partition WebSocket servers by user ID - Message deduplication with client-side UUID - Offline delivery: push to queue, deliver on reconnect

Distributed File Storage (S3-like)

// Dropbox, S3 Components: - Metadata service: { file_id, user_id, chunks[], size } - Chunk servers: store fixed-size file chunks - Load balancer: distributes uploads/downloads Upload flow: 1. Client splits file into 4MB chunks 2. For each chunk: hash → check if exists (dedup) 3. Upload only new chunks 4. Update metadata with chunk list Sync: - Client watches local folder for changes - Delta sync: only upload changed chunks - Versioning: keep chunk history
